# Benchmark harness

This directory defines a **comparison protocol** and ships the OSQP /
Clarabel adapters plus a harness that implements it. Published results live
under `results/YYYY-MM/`.

For single-solver smoke timings (Ledge only), see
[`../docs/SMOKE_TIMINGS.md`](../docs/SMOKE_TIMINGS.md). Those numbers are
install/scale sanity checks, not competitive claims.

## Microbenchmarks (regression tracking, not comparisons)

Criterion benches cover the reduced factorization, one x-update, and an
end-to-end synthetic solve. Run them all with one command:

```bash
cargo bench -p ledge-core --features bench-internals
```

The `bench-internals` feature exposes doc(hidden) solver hooks for timing and
must never be enabled by downstream users. Cloud CI only compiles the benches;
compare numbers on a fixed local machine (criterion keeps its baseline under
`target/criterion/`).

## Run a synthetic instance

```bash
cargo run --release -p ledge-portfolio --example synthetic -- \
  --n 500 --k 10 --inequalities 4 --seed 42
```

With a fixed `SyntheticConfig`, matrices, objectives, and the feasible
reference are generated by the built-in SplitMix64 + Box–Muller path (no
third-party RNG version skew). Output fields: status, objective, primal/dual
KKT residuals, iterations, and solver-reported wall time.

## Comparison protocol

Every published cross-solver number must satisfy all of these rules:

1. Use the same `QpProblem` data (no re-sampling);
2. Document the \(0.5x^\mathsf{T}Qx+q^\mathsf{T}x\) convention conversion;
3. Use the same primal warm start; record if dual warm start is unsupported;
4. Report each solver's own termination status — never rewrite iteration limit
   as solved;
5. Re-check returned primal/dual vectors with Ledge's independent `check_kkt`;
6. Split setup / first solve / repeated warm solves; report machine, commit,
   and build mode;
7. Repeat at least 10 times and publish all samples or median + quantiles —
   never only the fastest trial.

Until those rules are met for a claim, the root README must not repeat it.

## Comparison adapters (`adapters/`)

The workspace crate `ledge-bench-adapters` implements the protocol. OSQP and
Clarabel hide behind **non-default** cargo features, so default builds carry
no extra native dependency and never fail when those solvers are absent:

```bash
# Cross-check adapters against Ledge on small instances:
cargo test -p ledge-bench-adapters --features osqp,clarabel

# Full protocol run over the smoke matrix (writes CSV + Markdown):
cargo run --release -p ledge-bench-adapters --features osqp,clarabel \
  --bin compare -- --out benchmarks/results/YYYY-MM --repeats 10 \
  --rolling-steps 10 --commit "$(git rev-parse --short HEAD)"
```

The harness runs every smoke-matrix instance twice: the smooth base problem
and an `-l1` variant with proportional turnover costs anchored at the shared
primal start (`--l1-bps`, default `10` bps per asset; `0` disables the
variants). Ledge solves L1 with its prox block; external solvers receive the
documented epigraph reformulation (`n` extra variables, `2n` inequality
rows) with multipliers mapped back for re-verification.

Each external solver receives the shared instance in **two documented
formulations** (`adapters/src/convert.rs`):

- `dense-Q` — `Q = FΩF' + diag(d)` materialized densely, which is what a
  general QP solver sees when the user does not exploit factor structure;
  capped at `--dense-max-n` (default 1000) because cost and memory grow
  quadratically.
- `lifted` — `k` auxiliary variables `y = Ω^{1/2}F'x` with a sparse
  objective, the standard manual reformulation a sophisticated user would
  write. This is the fair baseline for cross-solver latency claims.

All three solvers minimize `0.5 x'Qx + q'x` natively, so no objective
rescaling occurs. Native statuses are recorded verbatim; duals are mapped
back into Ledge's sign convention and every returned point is re-verified
with `check_kkt` on the original data. Clarabel does not accept warm starts
(interior point); the harness records that instead of working around it.

`ledge_core::ComparisonSolver` / `BenchmarkRunner` remain as a minimal
single-solve hook used by the `synthetic` example; the protocol harness in
`adapters/src/protocol.rs` supersedes them for published comparisons.
