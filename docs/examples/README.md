# Examples

Runnable, end-to-end examples that are closer to production workloads than
the API snippets in the crate/package READMEs. Published numbers are
regenerated by running the scripts; each section records the build and
machine they came from.

## `rolling_backtest.py` — rolling rebalance with warm starts (roadmap 2.7)

A monthly momentum backtest on a public-data-style universe: 300 assets, a
market + 10-sector + momentum factor model (12 factors), 36 months of
generated returns, and 24 monthly rebalances driven by a 12-1 momentum
signal. Each date trades against the previous portfolio with exact 10 bps
proportional transaction costs (roadmap 2.1) and minimizes active risk
against a cap-weight-style benchmark (roadmap 2.6), long-only with 2%
position caps.

The same 24 dates are solved twice:

- **cold** — a fresh `solve_mean_variance_factor(...)` per date, paying
  equilibration, the SMW-reduced factorization, and un-warm-started ADMM
  every time;
- **rolling** — one `PortfolioProblem.sequence()`, where the factorizations
  are built once, per-date data arrives as `solve_next(expected_returns=...,
  previous_weights=...)`, and warm starts chain automatically.

![Rolling sequence data updates, warm starts, and factorization cache](../assets/rolling-rebalance.svg)

Run it from the repository root (needs the Python package built, e.g.
`maturin develop --release` in `python/`):

```bash
python docs/examples/rolling_backtest.py
```

### Published warm-start numbers

One run, seeded (`SEED = 7`), release build (`maturin develop --release`),
Python 3.12, Linux x86-64, 4-core Intel Xeon, 2026-07-22:

| | cold (fresh problem per date) | rolling (sequence) |
|---|---|---|
| iterations per warm date (mean, dates 2–24) | 79 | 37 (**2.1x fewer**) |
| wall time per warm date (mean, dates 2–24) | 1.71 ms | 0.91 ms (**1.9x faster**) |
| reduced factorizations across 24 dates | 24 | 2 |

Notes on reading the table:

- Date 1 is excluded from the means: the first solve of a sequence is cold
  by construction, so the rolling pass has no advantage there.
- The rolling pass built 2 factorizations rather than 1 because adaptive ρ
  rescaled once during the run; the cache is keyed by ρ, so later dates
  reuse both entries instead of refactorizing.
- Per-date one-way turnover under the 10 bps L1 costs settles around
  9–17%, printed per date by the script — the no-trade region is doing
  real work (compare `l1_turnover_costs=0.0` to see it disappear).
- At n=300 a single cold solve is already ~2 ms, so the absolute savings
  look small; iteration count (2.1x) is the machine-independent signal,
  and the wall-time gap widens with n (see `benchmarks/results/`).

The Rust equivalent of this workflow is `crates/ledge/examples/sequence.rs`
(`cargo run --example sequence`).
